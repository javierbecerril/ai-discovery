# Story Template

Reference format for all stories produced by `story-writer`.
Copy this format for each story. Stories can live in a single `STORIES.md` file or as individual files.

---

## STORY-[ID]: [Title]

- Persona: [which user type — from PROJECT_CONTEXT.md stakeholders]
- Story: As a [persona], I want [capability] so that [outcome].
- Context: [why this matters; include at least one direct quote as evidence from INSIGHT_REGISTRY.md]
- Acceptance Criteria:
  - [ ] [testable condition — specific, not vague]
  - [ ] [testable condition]
- Design Reference: [mockup file name, link, or description — or "none"]
- Related Stories: [STORY-XX, STORY-YY — or "none"]
- Theme: [THEME-XX from INSIGHT_REGISTRY.md]
- Priority: [High / Medium / Low]
- Needs clarification: [yes / no — if yes, state exactly what is unclear]

---

## Story Writing Notes

- One story = one deliverable chunk engineering can complete independently.
- Context is not optional — evidence quotes give engineering the "why."
- Acceptance criteria must be testable: avoid "it should feel better" — write "user can complete X in Y steps."
- Design references should point to the most specific available asset (a specific screen, not a whole Figma file).
- If a story needs clarification, still write it — flag the gap rather than skipping.
